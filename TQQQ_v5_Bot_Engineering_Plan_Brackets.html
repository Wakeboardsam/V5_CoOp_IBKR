Project Overview & Decisions

This document is the authoritative engineering specification for TQQQ Grid Bot v5. It is written to be passed directly to Google Jules as a series of structured pull request prompts. All architectural decisions are locked in below — Jules should not improvise on structure.

✓ Architecture Decisions — Locked All decisions below are confirmed and final. Jules should implement exactly as specified.

Decision 1: IBKR Connectivity via ib_insync

The bot connects to IBKR using ib_insync, a modern async-friendly Python wrapper around IBKR's official ibapi. It connects to a locally-running IB Gateway process (not TWS). The Gateway runs as a headless Java process inside the same Docker container, auto-logged in by IBC (IB Controller). This avoids any GUI requirement and is fully compatible with headless Docker environments.

Decision 2: Google Sheets as Source of Truth

The trading grid — buy prices, sell prices, and share quantities — lives entirely in a Google Sheet. The bot reads this sheet at startup and periodically to refresh the grid. It never writes its own prices or quantities. Only trade confirmation logs are written back to the sheet.

Decision 3: Broker Sandbox Pattern

All broker-specific code lives in brokers/ibkr/adapter.py. A base class brokers/base.py defines the interface. Switching to a new broker means only adding a new folder under brokers/ and changing active_broker in options.json. Zero changes to engine logic.

Decision 4: Paper Trading First

IBKR provides a full paper trading environment at localhost:7497 (vs live at 7496). The options.json flag "paper_trading": true toggles between them. All initial development and testing MUST use paper mode.

Decision 5: Extended Hours

TQQQ trades during extended hours (pre-market 4:00 AM–9:30 AM ET, after-hours 4:00 PM–8:00 PM ET). IBKR supports this via the outsideRth=True order flag. The bot will trade in these windows. TQQQ does NOT trade overnight (midnight–4 AM), so no futures are needed.

⚠ TQQQ Extended Hours Reality Check Pre-market liquidity on TQQQ can be thin with wide spreads. The grid logic must include a configurable max_spread_pct guard that skips order placement if the bid-ask spread exceeds threshold. Default: 0.5%.

Decision 6: Home Assistant Addon Constraints

The bot runs as a Home Assistant Addon — a Docker container managed by HA's Supervisor. The filesystem is ephemeral. All persistent config comes via options.json (HA's addon options). All code changes are deployed via GitHub. The container runs supervisord to manage both the IB Gateway process and the Python bot process simultaneously.

  • Boot: Config loaded from options.json → SheetInterface fetches grid → IBKRAdapter connects

Module Specifications

brokers/base.py — BrokerBase

# brokers/base.py — NEVER modify this file for broker-specific logic
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional, Callable

@dataclass
class OrderResult:
    order_id: str
    status: str          # 'submitted' | 'filled' | 'cancelled' | 'error'
    filled_price: Optional[float] = None
    filled_qty:   Optional[int]   = None
    error_msg:    Optional[str]   = None

class BrokerBase(ABC):

    @abstractmethod
    async def connect(self) -> bool: ...

    @abstractmethod
    async def disconnect(self): ...

    @abstractmethod
    async def get_price(self, ticker: str) -> float: ...

    @abstractmethod
    async def get_bid_ask(self, ticker: str) -> tuple[float, float]: ...

    @abstractmethod
    async def place_bracket_order(
        self, ticker: str, action: str,  # 'BUY' | 'SELL'
        qty: int, limit_price: float, profit_price: float,
        extended_hours: bool = True,
        on_fill: Optional[Callable] = None
    ) -> OrderResult: ...

    @abstractmethod
    async def cancel_order(self, order_id: str) -> bool: ...

    @abstractmethod
    async def get_open_orders(self) -> list[dict]: ...

    @abstractmethod
    async def get_positions(self) -> dict[str, int]: ...

brokers/ibkr/adapter.py — IBKRAdapter

# Key implementation notes for Jules:
# - Uses ib_insync.IB() as the connection object
# - All methods are async (ib_insync supports asyncio natively)
# - Paper port: 7497, Live port: 7496 (from config)
# - clientId: unique int per connection (use config value, default 1)
# - TQQQ contract: Stock('TQQQ', 'SMART', 'USD')
# - All limit orders must set order.outsideRth = True
# - All limit orders must set order.tif = 'GTC' (Good Till Cancelled)
# - Reconnect logic: exponential backoff, max 10 retries
# - Fill callback: subscribe to ib.orderStatusEvent

from ib_insync import IB, Stock, LimitOrder
from brokers.base import BrokerBase, OrderResult

class IBKRAdapter(BrokerBase):
    def __init__(self, host: str, port: int, client_id: int, paper: bool):
        self.host = host
        self.port = port          # 7497=paper, 7496=live
        self.client_id = client_id
        self.paper = paper
        self.ib = IB()
        self._contract = Stock('TQQQ', 'SMART', 'USD')

✓ Extended Hours Strategy All orders are placed as GTC Limit orders with outsideRth=True. This means an order placed at 9 PM will remain live when pre-market opens at 4 AM. The SpreadGuard is especially important during extended hours when spreads widen significantly.

Order Deduplication

The OrderManager prevents placing duplicate orders for the same grid level. If a BUY order for ROW_ID "BUY_01" is already open at IBKR, the engine will not submit another one even if the price re-triggers. Orders are only re-eligible after being filled or explicitly cancelled.

Broker Swap Guide

If IBKR doesn't work out and you need to move to another broker (Alpaca, Tradier, Tastytrade, etc.), follow these exact steps:

  1. Create brokers/<newbroker>/adapter.py implementing all BrokerBase methods
  2. Add any broker-specific config fields to config/schema.py
  3. Update options.schema.json for HA UI
  4. In options.json set "active_broker": "<newbroker>"
  5. No changes to engine/, sheets/, or main.py
✓ Alpaca is the Recommended Fallback Alpaca has a REST API with paper trading, supports extended hours on ETFs including TQQQ, has a clean Python SDK (alpaca-trade-api), and requires no locally-running Gateway process. The IBKRAdapter is ~200 lines — an AlpacaAdapter would be simpler at ~100 lines.
